While the spec is approved, here are two minor implementation notes for the builder to keep in mind:

1. **Default Logic via Exclusion:** To satisfy both the rule *"The default list should be failure, cancelled, timed_out"* AND the rule *"Events whose conclusion is empty/null/unknown are forwarded by default"*, the implementation for `config.conclusions === null` must be written as an **exclusion list** (e.g., `return !['success', 'skipped', 'neutral', 'action_required', 'stale', 'manual'].includes(normalizedConclusion)`). If implemented as an inclusion list, unknown values would accidentally be dropped.
2. **Explicit Empty Strings:** The existing `splitCommaList` helper in `lib/config.ts` uses `.filter(Boolean)`. This means if a user attempts to explicitly include the empty string in their filter (e.g., `--conclusions "failure,"`), it will be stripped out during parsing. This is an acceptable edge case since users opting into strict filtering usually want to drop empty states anyway, but the builder should be aware of this when writing tests.
