## Plan Review: Filter Notifications by Conclusion

### 1. Spec Coverage

The plan covers all spec requirements comprehensively. The mapping from spec success criteria to implementation steps is explicit, and the test scenarios (Phase 2) map 1:1 to the spec's 12 test scenarios. The reconciliation scoping exclusion is correctly carried forward. No phases exceed spec scope.

### 2. Technical Approach

The approach is architecturally sound. Placing `normalizeConclusion` and `isConclusionAllowed` in `webhook.ts` alongside the existing `isWorkflowAllowed` pattern is the right call — it mirrors a proven pattern the codebase already uses. The insertion point in the handler pipeline (between Step 5 workflow filter and Step 6 notification) is correct and verified against the actual code.

### 3. Critical Issues

#### Issue A: Existing tests WILL break in Phase 1 — plan doesn't account for this

The plan's Phase 1 success criteria says *"Existing test suite passes: `npm test`"* and *"No new tests in Phase 1. Regression-only."* But the behavior change **will break at least two existing integration tests**:

1. **`tests/integration.test.ts` line 169–183** — *"success event → 200 + notification with conclusion field"* — sends a `success` event and asserts a notification is produced. With the default filter active (no `conclusions` in `testConfig`), `success` is excluded. This test will fail.

2. **`tests/integration-gitlab.test.ts` line 176–179** — asserts that a `running` pipeline event produces a notification. `running` is in the default exclusion set. This test will fail.

The plan needs to explicitly instruct the builder to either:
- Add `conclusions: ['all']` to these test configs (to preserve pre-filter behavior for unrelated tests), or
- Update the test assertions to match the new default behavior

This is a Phase 1 concern, not Phase 2.

#### Issue B: TypeScript compilation will fail — `Config` interface change not propagated to tests

Adding `conclusions: string[] | null` to the `Config` interface means **every test file that constructs a `Config` object inline** will fail `npm run build` with a missing-property error. Affected files:

| File | Line | Object |
|------|------|--------|
| `tests/integration.test.ts` | 33–43 | `testConfig` |
| `tests/integration-gitlab.test.ts` | 27–37 | `testConfig` |
| `tests/integration-gitea.test.ts` | 32–42 | `testConfig` |
| `tests/bootstrap.test.ts` | 7–19 | `makeConfig()` helper |

All need `conclusions: null` added. This **must** happen in Phase 1 to satisfy the *"Type-checks: `npm run build` produces no errors"* success criterion. The plan lists only production files for Phase 1; it should include these test files as mechanical updates.

#### Issue C: `envKeys` in `config.test.ts` needs `CONCLUSIONS`

The test's env save/restore mechanism (line 13–16) enumerates all env vars it touches. `CONCLUSIONS` must be added to this array or Phase 2's config tests will leak env state across test cases.
